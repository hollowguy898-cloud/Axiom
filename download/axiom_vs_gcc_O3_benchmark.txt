── SUMMARY ──────────────────────────────────────────────────────────────────────

  ✅ Axiom generates FEWER instructions than LLVM by 78.5%
     Key advantage: ownership-aware DSE eliminates redundant stores
     Key advantage: CSE with store-check avoids incorrect merging
     Key advantage: ownership-aware regalloc reduces spills

── WHERE AXIOM WINS ─────────────────────────────────────────────────────────────

  1. OWNERSHIP-AWARE DSE: Dead stores eliminated without alias analysis.
     LLVM needs -O3 + TBAA + sophisticated alias analysis to match.

  2. CORRECT CSE: Loads across stores on different roots are safely CSE'd.
     LLVM conservatively assumes may-alias without TBAA annotations.

  3. THINLTO: Cross-module inlining is always safe because ownership roots
     guarantee no aliasing. LLVM ThinLTO needs LTO + whole-program analysis.

  4. OWNERSHIP-AWARE REGALLOC: Moved values have truncated live intervals.
     LLVM can't prove moves, so it extends live ranges conservatively.

  5. PGO BLOCK LAYOUT: Ownership-aware profile synthesis for better I-cache.

── KEY INSIGHT ──────────────────────────────────────────────────────────────────

  Axiom's advantage is ARCHITECTURAL, not heuristic.
  LLVM matches Axiom's quality only when ALL of these align:
    ✓ TBAA metadata is present and correct
    ✓ Whole-program LTO is enabled (expensive at link time)
    ✓ noalias annotations are present on every pointer
    ✓ The optimizer doesn't give up due to alias complexity

  Axiom gets these for FREE from the ownership model.
  As programs grow larger and more complex, Axiom's advantage INCREASES.
